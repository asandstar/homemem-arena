# HomeMem Arena：产品、游戏与研究设计基线

> 状态：当前主基线（Living Document）  
> 最近更新：2026-07-10  
> 适用范围：产品决策、关卡设计、工程实现、实验设计、QA 验收  
> 当其他文档与本文件冲突时，以本文件和实际代码为准；历史 QA/试玩报告只代表生成当日的快照。

## 1. 产品定义

HomeMem Arena 是一个以“回声屋”为玩家叙事外壳、以“具有限制记忆的家务机器人代理”为研究对象的 3D 长程任务游戏。

它必须同时满足三项要求：

1. **工程正确**：状态变化可解释，关卡可完成，输入方式结果一致，数据可复现。
2. **游戏好玩**：目标清楚、决策有张力、反馈及时、失败可学习、熟练后能进入心流。
3. **研究有效**：玩家确实需要使用记忆，环境扰动可控，行为数据完整，指标能回答长期记忆研究问题。

任何只满足其中一两项的功能都不算完成。例如：

- 只有可爱的叙事、但目标逻辑无法完成，不算游戏功能完成；
- 只有事件日志、但不同输入方式记录不同数据，不算研究采集完成；
- 只有科研字段、但玩家不需要记忆也能通关，不构成有效记忆实验。

## 2. 双层世界观

| 层次 | 面向对象 | 表达 |
|---|---|---|
| 玩家叙事层 | 普通玩家 | 玩家扮演实习家政小精灵“小橡”，在回声屋对抗四只记忆小妖 |
| 系统/研究层 | 研究者、数据导出 | 小橡对应受控机器人代理 `MEM-07`；记忆小妖对应环境扰动、状态变化和干扰变量 |

两层描述的是同一件事，不是两个主角：

- “小橡只能记住 3 样东西” = `MEM-07` 只有 3 个工作记忆槽；
- “钥匙猫把钥匙推走” = 空间记忆失效与重新验证实验；
- “餐盘精混淆脏净状态” = 物体状态记忆与相似物体辨别实验；
- “袜子幽灵移动衣物/篮子” = 多目标追踪、计数与空间更新实验；
- “早餐闹钟要求固定流程” = 程序记忆、时序记忆和长程阶段任务实验。

玩家界面优先使用“小橡/回声屋/记忆小妖”；研究数据、诊断报告和论文描述可以使用 `MEM-07`、agent、disturbance、probe 等术语，但应提供映射说明。

## 3. 三项产品支柱

### 3.1 工程正确性

不可妥协的工程契约：

- 所有交互入口（鼠标、键盘、未来的触屏/手柄）必须进入同一条 Game Command 管线。
- 一次有效命令必须原子地产生：领域状态变化、步数变化、事件记录、记忆影响、目标检查和反馈。
- 目标分为“历史里程碑”和“终局约束”。历史里程碑达成后不会因后续归位而撤销；终局约束必须在结束时仍成立。
- 关卡必须有显式生命周期：`idle → briefing → playing → probing → analyzing → result`。
- 只有 `playing` 状态允许计时、混乱增长和脚本事件触发。
- 随机事件必须由 session seed 驱动；相同版本、任务、seed 和命令序列应产生相同结果。
- GameStore 是当前世界状态的真值源；Session Recorder 是事件历史的真值源；二者不得各自实现一套相互矛盾的业务规则。

### 3.2 游戏性与心流

每个关卡应形成以下节奏：

1. **理解**：10–20 秒内理解目标和当前记忆限制。
2. **观察**：建立房间、物体、状态和容器的初始心智地图。
3. **计划**：决定先拿什么、记什么、是否锁定或更新记忆。
4. **执行**：移动、开容器、拾取、放置，获得即时反馈。
5. **扰动**：环境改变已知信息，迫使玩家验证或更新记忆。
6. **恢复**：玩家通过重新观察和记忆管理重新掌控局面。
7. **精通**：更少回访、更高 Combo、更低混乱、更准确 Probe。

挑战应主要来自有意义的认知取舍，而不是：

- 看不清关键物体；
- 不知道哪个键可用；
- 碰撞或门洞失灵；
- 随机事件无预警地连续触发；
- 目标文案与判定条件不一致；
- 阅读简报时计时器已经运行。

当前心流辅助基线：

- HUD 始终突出一个“当前专注”目标，降低同时追踪多个目标的工作记忆负担；
- 自上次目标进展起 20 秒无进展时，提示玩家只聚焦当前目标；
- 45 秒无进展时，给出与空间/物体/时序/程序记忆类型对应的策略提示，但不直接给答案；
- 任一目标取得进展后，提示等级和旧提示立即清零；
- 每次辅助都写入 `flow_intervention`，包含目标、等级和停滞时长。

`flowInterventionCount`、`longestGoalGapMs` 和 `actionSuccessRate` 只是行为代理指标，不能单独证明玩家进入心流。正式试玩还需在关卡后收集简短主观量表（如挑战—技能匹配、目标清晰、控制感、时间感变化）并与完成率、目标间隔和辅助触发次数联合分析。

### 3.3 研究价值

核心研究问题：

1. 有限工作记忆容量如何影响长程家务任务的完成策略？
2. 环境状态变化后，玩家何时会重新观察、更新或继续相信过期记忆？
3. 空间、物体、时序、程序四类记忆错误如何映射到可观察的失败行为？
4. 哪些策略能减少重复搜索、错误放置和无效路径？
5. 人类玩家形成的策略能否转化为家务机器人记忆策略或 benchmark policy？

研究设计原则：

- **可操作化**：每个研究概念必须对应可记录变量，而不只是一段文案。
- **可对照**：扰动强度、记忆槽数量、时间压力等应可配置，支持 A/B 或消融实验。
- **可复现**：保存任务版本、场景版本、seed、构建版本和实验条件。
- **最小混杂**：不要让操作困难、视觉搜索困难掩盖真正的记忆困难。
- **原始数据优先**：保留事件和状态变化；派生指标可以重算，不能只保存最终分数。
- **隐私透明**：正式众包前必须说明采集字段、用途、保留期限和退出机制。

## 4. 标准游戏与数据流程

```text
任务选择
  → 欢迎/简报（暂停）
  → 游戏执行（记录 observation / command / state delta / memory）
  → 达成终局条件或失败条件
  → 记忆 Probe（停止改变场景）
  → 统一计算指标和失败模式
  → 结果反馈
  → 导出/上传带版本的 Session
```

结果页不得在 Probe 之前 finalize Session。刷新、直接访问结果路由或重新游玩时，应有明确的恢复或重定向行为。

## 5. 关卡设计矩阵

| 顺序 | 关卡 | 主要记忆能力 | 核心扰动 | 玩家学习目标 |
|---|---|---|---|---|
| 1 | 餐桌混乱 | 物体状态、程序、空间 | 相似杯子、餐盘精把物品放回 | 区分状态、理解拾取/放置、第一次更新记忆 |
| 2 | 出门大作战 | 空间、物体 | 钥匙被移动、手机发声 | 跨房间规划、保存关键位置、处理过期记忆 |
| 3 | 洗衣幽灵 | 计数、物体、空间、时序 | 衣物移动、篮子扰动、额外物体 | 多目标跟踪、分类、重新计数 |
| 4 | 早餐时间循环 | 程序、时序、空间、物体 | 麦片移动、容器关闭、牛奶时间压力 | 执行“准备→使用→归位→关闭”的多阶段流程 |

难度递增优先通过以下参数完成：

- 房间数量和路径长度；
- 同时活跃的目标数量；
- 相似物体和状态辨别要求；
- 扰动发生的时机与可见性；
- 记忆槽与需要跟踪的信息量之比；
- 阶段数量和顺序约束；
- 合理的软时间压力。

## 6. 计分、失败与反馈

分数用于强化好策略，但不能替代研究指标。

推荐分解：

- 正确放置与阶段完成：基础分；
- 连续正确操作：Combo；
- 更新过期记忆、减少回访：策略奖励；
- 错误放置、无效回访、忘关容器：可解释惩罚；
- 剩余时间：小比例奖励，避免玩家只追求速度；
- Probe 准确率：单独展示，不混入任务操作分。

失败条件必须可解释并在结束画面中指向具体行为：

- 超时；
- 混乱过载；
- 关键对象进入不可恢复状态；
- 流程顺序进入不可恢复分支；
- 玩家主动终止。

失败后应支持快速重试，并保留“上次为什么失败”的一条可执行建议。

## 7. Session 最小研究契约

正式可用于分析的 Session 至少包含：

```text
schema_version, app_version, task_version, scene_version, seed
session_id, participant_id(匿名/可选), condition_id
start_time, end_time, lifecycle_status
commands, observations, agent_pose_trace(降采样)
object_state_changes, container_state_changes, scripted_events
memory_writes, memory_updates, memory_invalidations
goal_milestones, terminal_state
probe_questions, probe_answers, reaction_times
raw_outcome, derived_metrics, analysis_version
```

数据质量检查：

- action 数量与命令数量一致；
- 每个状态变化可追溯到玩家命令或脚本事件；
- Probe 在场景冻结后发生；
- 所有时间戳单调且以统一时钟为基准；
- 关卡结束后再计算最终指标；
- 导出前通过 schema 校验；
- 随机扰动可由 seed 重放。

## 8. 指标分层

### 游戏表现指标

- 是否完成、用时、得分、Combo、混乱峰值；
- 正确/错误放置；
- 重复搜索、无效房间回访；
- 路径长度、动作效率、阶段完成时间。

### 记忆行为指标

- 记忆槽使用率、覆盖率、锁定率；
- 过期记忆数量、发现延迟、更新延迟；
- 依赖过期记忆造成的错误；
- 各记忆类型的 Probe 正确率和反应时间。

### 研究派生指标

- memory effective rate；
- disturbance recovery cost；
- unnecessary revisit ratio；
- memory-to-action latency；
- procedure deviation count；
- observation efficiency。

派生指标必须记录定义和 `analysis_version`，避免算法调整后无法比较旧数据。

## 9. 验收门槛

### 可玩版本

- 4 个关卡都能从简报走到 Probe 和结果；
- 鼠标与键盘产生相同的领域结果；
- 没有不可完成目标或不可恢复软锁；
- 计时、暂停、失败和重试逻辑一致；
- 新玩家无需阅读外部说明即可完成第一关。

### 有趣版本

- 目标清楚但最优路线不是完全显然；
- 扰动有提示、有恢复窗口，不是纯随机惩罚；
- 玩家能理解记忆槽的价值，并主动做保存/覆盖/更新决策；
- 同一关重复游玩可通过更好策略提高表现；
- 试玩中能观察到“理解→紧张→恢复→精通”的节奏。

### 研究版本

- Session 契约完整且通过 schema 校验；
- seed 可重放；
- Probe 与指标在 finalize 前完整写入；
- 实验变量、版本和条件可追踪；
- 已完成知情说明、隐私策略和至少一轮内部数据质量审计。

## 10. 当前优先级

### P0：恢复正确闭环

1. 增加关卡生命周期与暂停语义。
2. 统一所有交互入口的命令/记录管线。
3. 实现真实 `timeLimit`。
4. 修复早餐关多阶段目标模型。
5. 调整流程为“游戏 → Probe → 分析 → 结果”。
6. 修复脚本事件的确定性、状态清理和 Session 记录。
7. 修复重玩与直接路由行为。

### P1：建立研究可信度

1. Session Recorder 与状态 delta。
2. 版本、seed、schema 校验和本地持久化。
3. 完整 pose/observation/memory/probe 记录。
4. 可重算、带版本的指标管线。
5. E2E 数据完整性测试。

### P2：提高心流与可扩展性

已完成心流机制基线：单一当前目标、20/45 秒渐进式停滞救援、辅助事件记录和三个过程代理指标。

1. 根据真人试玩数据调节时间、扰动、辅助阈值和评分曲线。
2. 加入关卡后主观心流短量表，验证代理指标是否与体验一致。
3. 优化新手引导、对象可读性和失败恢复。
4. 降低 React/Three.js 每帧状态更新成本。
5. 路由拆包、移动端/手柄支持和可访问性。
6. 建立关卡编辑与实验条件配置工具。
